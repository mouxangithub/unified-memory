## 1. Executive Summary
<!-- zh -->

**English:**

Unified Memory v4.0 is a ground-up re-architecture targeting production-scale multi-agent deployments. Where v3.8.x delivered core functionality with hybrid search, WAL durability, and evidence chains, v4.0 addresses the hard scaling problems: persistent indexes that survive restarts, truly incremental updates that eliminate full rebuilds, distributed rate limiting for multi-agent coordination, and a multi-tenant team-space model that replaces ad-hoc scope prefixes.

The central theme of v4.0 is **incremental persistence**: every data structure that was rebuilt in-memory at startup is now persisted to SQLite. Every index that was recomputed on every search is now updated transactionally on every write. The result is O(1) startup from SQLite, O(Δ) incremental updates vs. O(n) full rebuilds, and sub-millisecond scope filtering via persistent B-tree indexes.

**Key changes vs. v3.8.x:**
- SQLite-first storage (WAL → SQLite WAL table, JSON files → SQLite memories table)
- Incremental BM25 index with per-term delta updates and persistent posting lists
- Incremental LanceDB vector indexing with write-behind batching (500ms flush)
- Persistent scope index in SQLite B-tree (survives restarts, no rebuild)
- Multi-tenant team spaces replacing `team:{id}:` scope prefixes
- Per-agent and per-team rate limiting with SQLite-based distributed counters
- Evidence chain store in SQLite with B-tree index for O(log n) TTL trim
- Version history in SQLite with B-tree on `(memory_id, version)` for O(log n) conflict detection
- API versioning strategy with `/api/v4/` prefix and deprecation headers

**中文摘要:**

v4.0 是面向生产级多 Agent 部署的彻底重构版。v3.8.x 实现了混合搜索、WAL 持久化和证据链等核心功能，v4.0 则解决了硬性扩展问题：持久化索引（重启不丢失）、真正增量更新（消除全量重建）、分布式限流（多 Agent 协调）和多租户团队空间模型。

v4.0 的核心主题是**增量持久化**：所有曾在内存中重建的数据结构现在都持久化到 SQLite；所有每次搜索时重新计算的索引现在都在每次写入时事务性更新。

## 2. v3.8.x Technical Debt Audit
<!-- zh -->

### 2.1 Critical (P1) Issues

#### P1-1: BM25 Index Rebuilds on Every Search
**File:** `src/bm25.js`
**Problem:** `_cachedIndex` is invalidated when memory count changes, but the cache key is only `(scope, count)`. When count changes between searches, a full rebuild of the inverted index is triggered from scratch via `getAllMemories()` — O(n) tokenization of every document on every search-triggering-write.

```javascript
// v3.8.x bm25.js — problematic cache invalidation
let _cachedIndex = null;
let _cachedMemCount = -1;  // only checked for equality
let _cachedScope = null;

async function bm25Search(query, scope, options = {}) {
  const allMemories = await getAllMemories();  // always loads ALL memories from disk
  if (allMemories.length !== _cachedMemCount) {
    _buildIndex(allMemories);  // full O(n) rebuild — every search that follows a write
  }
  // ...
}
```

**v4.0 Fix:** Incremental BM25 with persistent posting lists in SQLite. Writes go through `BM25Indexer.updateDocument(id, text, oldText)` which mutates the posting list in-place with O(Δtokens) cost. No full rebuild ever triggered by search. Startup loads pre-built posting lists from SQLite — O(1) metadata load, no rebuilding.

#### P1-2: Scope Index Lost on Every Restart
**File:** `src/scope.js`
**Problem:** `teamRegistry` Map and `TEAM_CONTEXT`/`AGENT_CONTEXT` are pure in-memory. No persistence. After restart, all team/agent metadata is lost.

```javascript
// v3.8.x scope.js — in-memory only, lost on restart
const teamRegistry = new Map();  // not persisted
const TEAM_CONTEXT = { current: null };  // not persisted
const AGENT_CONTEXT = { current: null }; // not persisted
```

**v4.0 Fix:** `scopes` SQLite table with B-tree indexes on `(scope_type, scope_id)`, `(agent_id)`, `(team_id)`. All scope metadata survives restart. Team/agent context restored from DB on startup.

#### P1-3: 18 Modules Using Local Patterns Instead of Central Storage
**Files identified:** `agents/memory_agent.js`, `api/api.js`, `memory.js`, `graph/graph.js`, `graph/knowledge_card.js`, `memory_diff.js`, `cli/dedup_commands.js`, `decay/weibull_tools.js`, `core/usage_stats.js`, `revision_manager.js`, `cache_semantic.js`, `adaptive.js`, `cognitive.js`, `tier.js`, `organize.js`, `lanes.js`, `retrieval_tracer.js`, `index_builder.js`

**Problem:** Each module implements its own `getAllMemories()` / `getMemory()` pattern, creating 18 divergent code paths that bypass central storage transactions and WAL. Result: inconsistent reads, duplicate code, impossible atomicity across modules.

**v4.0 Fix:** Single `StorageGateway` class in `src/v4/storage-gateway.js`. All reads go through `StorageGateway.getMemories(filters)`. All writes go through `StorageGateway.write(memory)` which handles WAL + SQLite + vector index atomically. All 18 modules updated to use the gateway.

### 2.2 High Priority (P2) Issues

#### P2-1: Evidence Trim is O(n) Across All Entries
**File:** `src/evidence.js`
**Problem:** Evidence in `evidence.json` as flat JSON object keyed by `memory_id`. Global cap and TTL eviction scans ALL entries — O(n).

```javascript
// v3.8.x evidence.js — O(n) trim
function trimEvidence() {
  const all = JSON.parse(readFileSync(EVIDENCE_FILE));  // load ALL
  for (const [memoryId, chain] of Object.entries(all)) {
    chain.chain = chain.chain.filter(e => Date.now() - e.timestamp < TTL);
  }
}
```

**v4.0 Fix:** `evidence` SQLite table with B-tree index on `(memory_id, timestamp)`. TTL trim becomes:
```sql
DELETE FROM evidence WHERE timestamp < ?;
```
O(log n + m) via B-tree, not O(n) full scan.

#### P2-2: Scope Index Not Persistent
**(Same as P1-2 — resolved above)**

#### P2-3: Version History Trim is O(n)
**File:** `src/revision_manager.js`
**Problem:** `revisions.json` is a flat array. Trim requires loading all revisions, filtering by memory_id, re-writing entire file.

**v4.0 Fix:** `revisions` SQLite table with composite B-tree on `(memory_id, version)`. Trim:
```sql
DELETE FROM revisions WHERE memory_id = ? AND version NOT IN (
  SELECT version FROM revisions WHERE memory_id = ? ORDER BY version DESC LIMIT ?
);
```
O(log n) via index seek + O(k) delete.

#### P2-4: No Incremental Vector Indexing
**File:** `src/vector_lancedb.js`
**Problem:** LanceDB table appended without deduplication. No dirty-flag. Updated memories leave stale vectors.

**v4.0 Fix:** `vector_index_meta` SQLite table tracks `memory_id`, `vector_version`, `dirty` flag. On write, only `dirty = 1` vectors are re-indexed. LanceDB upsert uses `memory_version` filter — only updates if SQLite version > LanceDB version.

#### P2-5: Write-Behind Has No Atomicity Guarantee
**File:** `src/storage.js`
**Problem:** `_pendingMemories` flushes to JSON file before WAL is written. Crash between write and WAL = data loss.

```javascript
// v3.8.x — WAL written AFTER file save
async _flushPendingWrite() {
  await writeFile(tmpPath, data);  // ← crash here = data loss
  await rename(tmpPath, lockPath);
  if (global._walInit) logWriteOp({ type: 'save', count: memories.length });  // WAL after
}
```

**v4.0 Fix:** All writes go through a single SQLite transaction. WAL is a SQLite table (`wal_entries`) in the same DB file. SQLite's WAL mode ensures durability. Write: WAL entry (uncommitted) → apply to table → mark WAL committed.

#### P2-6: No Version Conflict Detection at Storage Layer
**File:** `src/storage.js`
**Problem:** No optimistic locking. Concurrent writes silently overwrite each other.

**v4.0 Fix:** `memories.version` column with optimistic locking:
```sql
UPDATE memories SET content = ?, version = version + 1, updated_at = ?
WHERE id = ? AND version = ?;
-- If affected rows = 0, conflict → throw ConflictError, retry with fresh version.
```

#### P2-7: No Per-Agent or Per-Team Rate Limiting
**Problem:** No rate limiting at all. A single agent can flood the system unlimited.

**v4.0 Fix:** `rate_limits` SQLite table with sliding-window counter per `(agent_id, operation_type, window_start)`. Atomic increment via `INSERT ... ON CONFLICT DO UPDATE`. Configurable per agent/team via `agent_config` / `team_config` tables.

### 2.3 Missing Capabilities (P3)

#### P3-1: No Multi-Tenant Team Spaces
**Problem:** Team isolation via `team:{id}:` prefix — not a real tenant model. No ACL, no quota, no per-team config.

**v4.0 Fix:** `team_spaces` table as first-class tenants. Each has own namespace, ACL, rate limits, quota. Memories carry `team_space_id` FK. Members have explicit roles (owner/member/viewer).

#### P3-2: No API Versioning Strategy
**Problem:** All endpoints at `/api/` without version prefix. No deprecation policy.

**v4.0 Fix:** All v4.0 endpoints under `/api/v4/`. Response headers: `Deprecation: true`, `Sunset: <date>` for deprecated endpoints. `GET /api/v4/version` returns server version.

#### P3-3: LanceDB Index Not Checkpointed
**Problem:** Vector index with no checkpoint. Corruption requires full re-embedding of all memories.

**v4.0 Fix:** `vector_checkpoint` table tracks last durable LSN. On startup, compare with SQLite checkpoint. If mismatch, re-index only dirty vectors — O(Δ), not O(n).

## 3. v4.0 Architecture

### 3.1 Design Principles

| Principle | Rationale |
|-----------|-----------|
| **SQLite as single source of truth** | Replaces JSON files + evidence.json + revisions.json + WAL files. One DB file, one transaction log. |
| **Incremental by default** | No full rebuilds triggered by reads. All write paths update indexes in-place via delta updates. |
| **Persistent indexes** | All indexes (BM25 posting lists, scope B-trees, vector metadata) survive process restarts. |
| **Multi-tenant from day one** | `team_space_id` FK on every record. No retrofitting later. |
| **Zero-copy read paths** | Hot-path reads go directly from SQLite with covering indexes — no JSON parse. |
| **Optimistic locking everywhere** | Every mutable entity has a `version` column. Concurrent writes detected and retried. |

### 3.3 Persistent Scope Index with Incremental Updates

**v3.8.x problem:** `scope.js` teamRegistry Map in pure memory. Lost on restart. Scope filtering was O(n) scan.

**v4.0 solution:** SQLite `scopes` table with B-tree indexes.

```sql
CREATE TABLE scopes (
  id              TEXT PRIMARY KEY,       -- 'agent:{id}' or 'team:{id}' or 'user:{id}'
  scope_type      TEXT NOT NULL,          -- 'AGENT' | 'USER' | 'TEAM' | 'GLOBAL'
  parent_id       TEXT,                   -- team_id for TEAM members, NULL otherwise
  team_space_id   TEXT,                   -- FK to team_spaces.id
  name            TEXT,
  metadata        TEXT NOT NULL DEFAULT '{}',
  created_at      INTEGER NOT NULL,
  updated_at      INTEGER NOT NULL
);

CREATE INDEX idx_scopes_type ON scopes(scope_type);
CREATE INDEX idx_scopes_team_space ON scopes(team_space_id) WHERE team_space_id IS NOT NULL;
CREATE INDEX idx_scopes_parent ON scopes(parent_id) WHERE parent_id IS NOT NULL;
```

**Scope filter at search time (uses index, not full scan):**

```sql
-- TEAM scope: all members' memories within the team space
SELECT m.* FROM memories m
WHERE m.team_space_id = :teamSpaceId
  AND m.scope_type IN ('TEAM', 'GLOBAL');

-- USER scope: user's own + team spaces they belong to + global
SELECT m.* FROM memories m
WHERE (m.scope_type = 'USER' AND m.user_id = :userId)
   OR (m.scope_type = 'TEAM' AND m.team_space_id IN (
       SELECT team_space_id FROM team_members WHERE user_id = :userId
   ))
   OR m.scope_type = 'GLOBAL';

-- AGENT scope: strict isolation, only current agent's memories
SELECT m.* FROM memories m
WHERE m.agent_id = :agentId AND m.scope_type = 'AGENT';
```

### 3.4 Incremental BM25 (No Full Rebuild)

**v3.8.x problem:** BM25 index rebuilt from scratch on every search when memory count changed. O(n) tokenization on hot path.

**v4.0 solution:** Five persistent SQLite tables updated incrementally on every write.

```sql
-- Term dictionary: maps term → term_id
CREATE TABLE bm25_term_dict (
  term_id   INTEGER PRIMARY KEY AUTOINCREMENT,
  term      TEXT UNIQUE NOT NULL,
  df        INTEGER NOT NULL DEFAULT 0,  -- document frequency
  updated_at INTEGER NOT NULL
);

-- Posting list: (term_id, memory_id, tf, positions)
CREATE TABLE bm25_postings (
  term_id     INTEGER NOT NULL REFERENCES bm25_term_dict(term_id),
  memory_id   TEXT NOT NULL,
  tf          INTEGER NOT NULL DEFAULT 0,   -- term frequency in this document
  positions   TEXT,                          -- JSON array of positions (phrase queries)
  PRIMARY KEY (term_id, memory_id)
);

CREATE INDEX idx_postings_memory ON bm25_postings(memory_id);
CREATE INDEX idx_postings_term_tf ON bm25_postings(term_id, tf DESC);

-- Document metadata for length normalization
CREATE TABLE bm25_doc_meta (
  memory_id    TEXT PRIMARY KEY,
  doc_length   INTEGER NOT NULL DEFAULT 0,
  updated_at   INTEGER NOT NULL
);

-- BM25 global parameters (singleton, id=1)
CREATE TABLE bm25_config (
  id           INTEGER PRIMARY KEY CHECK (id = 1),
  k1            REAL NOT NULL DEFAULT 1.5,
  b             REAL NOT NULL DEFAULT 0.75,
  avgdl         REAL NOT NULL DEFAULT 0,
  doc_count     INTEGER NOT NULL DEFAULT 0,
  total_tokens  INTEGER NOT NULL DEFAULT 0,
  updated_at    INTEGER NOT NULL
);

INSERT INTO bm25_config (id, k1, b, avgdl, doc_count, total_tokens, updated_at)
VALUES (1, 1.5, 0.75, 0, 0, 0, 0);
```

**Incremental update algorithm (called on every memory write):**

```javascript
/**
 * Update BM25 posting lists incrementally for a memory write.
 * @param {string} memoryId - Memory ID
 * @param {string} newText - New memory content (tokenized)
 * @param {string|null} oldText - Previous content (for update operations, null for insert)
 */
async function updateBM25Index(memoryId, newText, oldText = null) {
  const newTokens = tokenize(newText);
  const oldTokens = oldText ? tokenize(oldText) : [];
  const newTokenCounts = countTokens(newTokens);  // Map<token, count>
  const oldTokenCounts = countTokens(oldTokens);

  const db = getV4DB();

  await db.transaction(async (tx) => {
    // Process added/new tokens
    for (const [token, tf] of newTokenCounts) {
      await tx.run(
        `INSERT INTO bm25_term_dict (term, df, updated_at) VALUES (?, 0, ?)
         ON CONFLICT(term) DO UPDATE SET updated_at = excluded.updated_at`,
        [token, Date.now()]
      );

      const termRow = await tx.get(`SELECT term_id, df FROM bm25_term_dict WHERE term = ?`, [token]);
      const existingTf = oldTokenCounts.get(token) || 0;
      const positions = JSON.stringify(getPositions(newTokens, token));

      await tx.run(
        `INSERT INTO bm25_postings (term_id, memory_id, tf, positions) VALUES (?, ?, ?, ?)
         ON CONFLICT(term_id, memory_id) DO UPDATE SET tf = excluded.tf, positions = excluded.positions`,
        [termRow.term_id, memoryId, tf, positions]
      );

      // Increment doc frequency if term is new to this document
      if (existingTf === 0) {
        await tx.run(
          `UPDATE bm25_term_dict SET df = df + 1, updated_at = ? WHERE term_id = ?`,
          [Date.now(), termRow.term_id]
        );
      }
    }

    // Process removed tokens (present in old but not in new)
    for (const [token, oldTf] of oldTokenCounts) {
      if (newTokenCounts.has(token)) continue;
      const termRow = await tx.get(`SELECT term_id FROM bm25_term_dict WHERE term = ?`, [token]);
      if (!termRow) continue;
      await tx.run(`DELETE FROM bm25_postings WHERE term_id = ? AND memory_id = ?`, [termRow.term_id, memoryId]);
      await tx.run(`UPDATE bm25_term_dict SET df = MAX(0, df - 1), updated_at = ? WHERE term_id = ?`, [Date.now(), termRow.term_id]);
    }

    // Update doc length metadata
    const oldDocLen = oldText ? oldTokens.length : 0;
    const newDocLen = newTokens.length;
    await tx.run(
      `INSERT INTO bm25_doc_meta (memory_id, doc_length, updated_at) VALUES (?, ?, ?)
       ON CONFLICT(memory_id) DO UPDATE SET doc_length = excluded.doc_length, updated_at = excluded.updated_at`,
      [memoryId, newDocLen, Date.now()]
    );

    // Update global average document length incrementally
    if (!oldText) {
      // Insert: new document
      await tx.run(
        `UPDATE bm25_config SET doc_count = doc_count + 1, total_tokens = total_tokens + ?,
         avgdl = (total_tokens + ?) / (doc_count + 1), updated_at = ? WHERE id = 1`,
        [newDocLen, newDocLen, Date.now()]
      );
    } else {
      // Update: document replaced
      await tx.run(
        `UPDATE bm25_config SET total_tokens = total_tokens - ? + ?,
         avgdl = (total_tokens - ? + ?) / doc_count, updated_at = ? WHERE id = 1`,
        [oldDocLen, newDocLen, oldDocLen, newDocLen, Date.now()]
      );
    }
  });
}
```

**Search algorithm (no rebuild — uses pre-built posting lists):**

```javascript
async function bm25Search(query, options = {}) {
  const { limit = 20 } = options;
  const queryTokens = tokenize(query);
  if (queryTokens.length === 0) return [];

  const config = await db.get(`SELECT k1, b, avgdl, doc_count FROM bm25_config WHERE id = 1`);

  // Get term_ids for query tokens (only terms in dictionary)
  const termIds = await db.all(
    `SELECT term_id FROM bm25_term_dict WHERE term IN (${queryTokens.map(() => '?').join(',')})`,
    queryTokens
  );
  if (termIds.length === 0) return [];

  const tidList = termIds.map(t => t.term_id).join(',');

  // BM25 scoring via SQL — no in-memory index rebuild
  const results = await db.all(`
    SELECT
      p.memory_id,
      SUM(
        log((:docCount - t.df + 0.5) / (t.df + 0.5) + 1)
        * (:tf * (:k1 + 1)) / (:tf + :k1 * (1 - :b + :b * dm.doc_length / :avgdl))
      ) AS bm25_score
    FROM bm25_postings p
    JOIN bm25_term_dict t ON p.term_id = t.term_id
    JOIN bm25_doc_meta dm ON p.memory_id = dm.memory_id
    WHERE p.term_id IN (${tidList})
    GROUP BY p.memory_id
    ORDER BY bm25_score DESC
    LIMIT ?
  `, { docCount: config.doc_count, k1: config.k1, b: config.b, avgdl: config.avgdl, tf: 1 });

  return results;
}
```

**Tokenization helper (consistent with v3.8.x):**

```javascript
function tokenize(text) {
  if (!text) return [];
  const tokens = [];
  text = text.replace(/<[^>]+>/g, ' ');
  const english = text.match(/[a-zA-Z_]+/g) || [];
  for (const word of english) {
    const lower = word.toLowerCase();
    if (lower.length > 1) tokens.push(lower);
  }
  const chinese = text.match(/[\u4e00-\u9fa5]/g) || [];
  const stopChars = new Set(['的', '了', '在', '是', '我', '有', '和', '就', '不', '人', '都', '一', '上', '也', '很', '到', '说', '要', '去', '你', '会', '着', '看', '好', '自', '这', '那', '么']);
  for (const char of chinese) {
    if (!stopChars.has(char)) tokens.push(char);
  }
  return tokens;
}

function countTokens(tokens) {
  const counts = new Map();
  for (const t of tokens) counts.set(t, (counts.get(t) || 0) + 1);
  return counts;
}

function getPositions(tokens, targetToken) {
  const positions = [];
  for (let i = 0; i < tokens.length; i++) {
    if (tokens[i] === targetToken) positions.push(i);
  }
  return positions;
}
```

### 3.5 Incremental Vector Indexing

**v3.8.x problem:** LanceDB appended without deduplication. No dirty-set. Full re-embedding required on restart.

**v4.0 solution:** SQLite dirty tracking + LanceDB with version-based upsert.

```sql
-- Vector index metadata: tracks which memories need re-embedding
CREATE TABLE vector_index_meta (
  memory_id      TEXT PRIMARY KEY,
  vector_version INTEGER NOT NULL DEFAULT 0,
  embedding      TEXT,                         -- base64 (for verification)
  embedding_model TEXT,
  indexed_at     INTEGER,
  dirty          INTEGER NOT NULL DEFAULT 1,    -- 1 = needs re-indexing
  updated_at     INTEGER NOT NULL
);

CREATE INDEX idx_vector_dirty ON vector_index_meta(dirty) WHERE dirty = 1;

-- Vector checkpoint: crash recovery anchor
CREATE TABLE vector_checkpoint (
  id              INTEGER PRIMARY KEY CHECK (id = 1),
  last_lsn        TEXT NOT NULL,
  last_indexed_at INTEGER NOT NULL,
  checkpoint_at   INTEGER NOT NULL
);

-- LanceDB table: memories_vectors
-- Schema: { id: memory_id (PK), scope: string, embedding: float[1536], text: string, memory_version: int }
```

**Write path (StorageGateway.write):**

```javascript
async function writeWithVectorIndexing(memory) {
  const db = getV4DB();
  const newVersion = (await db.get(`SELECT vector_version FROM vector_index_meta WHERE memory_id = ?`, [memory.id]))?.vector_version + 1 || 1;

  await db.transaction(async (tx) => {
    // Upsert memory to SQLite memories table (optimistic locking)
    const result = await tx.run(`
      INSERT INTO memories (id, content, scope_type, agent_id, user_id, team_space_id, category, importance, tier, vector_version, version, created_at, updated_at)
      VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, 1, ?, ?)
      ON CONFLICT(id) DO UPDATE SET
        content = excluded.content, vector_version = excluded.vector_version,
        version = version + 1, updated_at = excluded.updated_at
      WHERE version = (SELECT version FROM memories WHERE id = excluded.id)
    `, [memory.id, memory.content, memory.scope_type, memory.agent_id, memory.user_id,
        memory.team_space_id, memory.category, memory.importance, memory.tier,
        newVersion, Date.now(), Date.now()]);

    if (result.changes === 0) throw new ConflictError('Memory was modified by another writer');

    // Mark vector as dirty (needs re-indexing)
    await tx.run(`
      INSERT INTO vector_index_meta (memory_id, vector_version, dirty, updated_at)
      VALUES (?, ?, 1, ?)
      ON CONFLICT(memory_id) DO UPDATE SET
        vector_version = excluded.vector_version, dirty = 1, updated_at = excluded.updated_at
    `, [memory.id, newVersion, Date.now()]);
  });
}
```

**VectorIndexer write-behind (every 500ms):**

```javascript
class VectorIndexer {
  constructor() {
    this.dirtyBatch = [];
    this.flushInterval = setInterval(() => this.flushDirty(), 500);
  }

  async flushDirty() {
    if (this.dirtyBatch.length === 0) return;
    const batch = this.dirtyBatch.splice(0, 100);  // process up to 100

    const db = getV4DB();
    const memories = await db.all(`
      SELECT m.id, m.content, v.vector_version, v.embedding_model
      FROM memories m
      JOIN vector_index_meta v ON m.id = v.memory_id
      WHERE v.dirty = 1 AND m.id IN (${batch.map(() => '?').join(',')})
    `, batch);

    if (memories.length === 0) return;

    // Batch embedding generation (Ollama batch API)
    const embeddings = await embedBatch(memories.map(m => m.content));

    // LanceDB upsert with version check (only if LanceDB version < SQLite version)
    for (let i = 0; i < memories.length; i++) {
      const mem = memories[i];
      const emb = embeddings[i];
      await lanceTable.mergeInsert([{
        id: mem.id,
        scope: mem.scope_type,
        embedding: emb,
        text: mem.content.slice(0, 500),
        memory_version: mem.vector_version
      }]).where('memory_version < $1').params([mem.vector_version]).execute();
    }

    // Mark clean
    await db.run(`
      UPDATE vector_index_meta SET dirty = 0, indexed_at = ?,
      embedding = ?, updated_at = ?
      WHERE memory_id IN (${batch.map(() => '?').join(',')})
    `, [Date.now(), encodeBase64(embeddings[0]), Date.now(), ...batch]);
  }
}
```

**Rate limit check (atomic, single round-trip):**

```javascript
async function checkRateLimit(agentId, operationType) {
  const windowStart = Math.floor(Date.now() / 60000) * 60;
  const key = `agent:${agentId}:${operationType}`;

  const agentCfg = await db.get(
    `SELECT rate_limit_write, rate_limit_read, rate_limit_search FROM agent_config WHERE agent_id = ?`,
    [agentId]
  );
  const limit = agentCfg?.[`rate_limit_${operationType}`] ?? 60;

  const result = await db.transaction(async (tx) => {
    const current = await tx.get(
      `SELECT count FROM rate_limits WHERE key = ? AND window_start = ?`,
      [key, windowStart]
    );

    if (current && current.count >= limit) {
      return { allowed: false, remaining: 0, resetAt: windowStart + 60 };
    }

    await tx.run(`
      INSERT INTO rate_limits (key, window_start, count) VALUES (?, ?, 1)
      ON CONFLICT(key, window_start) DO UPDATE SET count = count + 1
    `, [key, windowStart]);

    const newCount = current ? current.count + 1 : 1;
    return { allowed: true, remaining: Math.max(0, limit - newCount), resetAt: windowStart + 60 };
  });

  if (!result.allowed) {
    throw new RateLimitError(
      `Rate limit exceeded for ${operationType}. Resets at ${new Date(result.resetAt * 1000).toISOString()}`
    );
  }
  return result;
}
```

**Rate limit status:**

```javascript
async function getRateLimitStatus(agentId) {
  const windowStart = Math.floor(Date.now() / 60000) * 60;
  const limits = await db.all(`
    SELECT key, count FROM rate_limits
    WHERE key LIKE 'agent:${agentId}:%' AND window_start = ?
  `, [windowStart]);

  return limits.reduce((acc, row) => {
    const [, , op] = row.key.split(':');
    acc[op] = { used: row.count, resetAt: windowStart + 60 };
    return acc;
  }, { write: { used: 0 }, read: { used: 0 }, search: { used: 0 } });
}
```

### 3.8 WAL to SQLite Migration

**v3.8.x problem:** WAL as `data/wal/*.wal.jsonl` text files. Recovery required parsing JSONL. WAL segments grew unbounded.

**v4.0 solution:** WAL entries in SQLite `wal_entries` table. SQLite's own WAL mode provides crash protection.

```sql
CREATE TABLE wal_entries (
  lsn           TEXT PRIMARY KEY,          -- '<timestamp>_<sequence_num>'
  operation     TEXT NOT NULL,            -- 'insert' | 'update' | 'delete'
  collection    TEXT NOT NULL,            -- 'memories' | 'evidence' | 'revisions'
  record_id     TEXT NOT NULL,
  payload       TEXT NOT NULL,            -- JSON of the record
  checksum      TEXT NOT NULL,            -- SHA-256 of payload
  agent_id      TEXT,
  team_space_id TEXT,
  committed     INTEGER NOT NULL DEFAULT 0,  -- 0=in-flight, 1=committed, 2=rolled back
  created_at    INTEGER NOT NULL
);

CREATE INDEX idx_wal_lsn ON wal

### 5.2 Automatic Migration (Recommended)

v4.0 auto-detects v3.8.x data on startup and runs migration in a single transaction:

```javascript
// src/v4/migrate-from-v3.js (pseudocode)
async function migrateFromV38() {
  const db = await openV4Database();
  
  await db.transaction(async () => {
    // Phase 1: memories.json → memories table
    const v38memories = JSON.parse(readFileSync('memory/memories.json'));
    for (const mem of v38memories) {
      await db.run(`
        INSERT OR REPLACE INTO memories 
        (id, text, category, tags, importance, scope_type, scope_id, 
         tier, pinned, created_at, updated_at, accessed_at)
        VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
      `, [mem.id, mem.text, mem.category, JSON.stringify(mem.tags || []),
          mem.importance, mem.scope || 'USER', null,
          mem.tier || 'HOT', mem.pinned ? 1 : 0,
          mem.created_at, mem.updated_at || mem.created_at,
          mem.accessed_at || mem.created_at]);
    }

    // Phase 2: evidence.json → evidence table
    const evidenceData = JSON.parse(readFileSync('memory/evidence.json'));
    for (const [memoryId, chainData] of Object.entries(evidenceData)) {
      for (const entry of chainData.chain || []) {
        await db.run(`INSERT INTO evidence ...`, [...]);
      }
    }

    // Phase 3: Rebuild BM25 posting lists from migrated memories
    const allMems = await db.all('SELECT id, text FROM memories WHERE deleted=0');
    for (const mem of allMems) {
      await BM25Indexer.indexDocument(mem.id, mem.text);
    }

    // Phase 4: Rebuild vector index metadata
    for (const mem of allMems) {
      if (mem.text) {
        await VectorIndexer.indexDocument(mem.id, mem.text);
      }
    }

    // Phase 5: Commit migration marker
    await db.run("INSERT INTO meta (key, value) VALUES ('migrated_from', 'v3.8')");
  });
}
```

# 集成对比

[English](./en/INTEGRATION_COMPARISON.md) | [概述](./README.md) | 中文

## 三种集成模式

Unified Memory 支持三种集成方式。根据你的需求选择：

| 模式 | 自动捕获 | 手动检索 | 复杂度 | 控制力 |
|------|:--------:|:--------:|:------:|:------:|
| **Hook + MCP** | ✅ | ✅ | 中等 | 高 |
| **仅 MCP** | ❌ | ✅ | 低 | 完全 |
| **仅 Hook** | ✅ | ❌ | 低 | 低 |

---

## 模式一：Hook + MCP（推荐）

**适用于：** 需要自动捕获同时又想具备搜索/管理记忆能力的多数用户。

```json
{
  "plugins": {
    "entries": {
      "unified-memory": {
        "hook": "enabled",
        "config": {
          "mcpEnabled": true,
          "autoExtract": true
        }
      }
    }
  }
}
```

### 优点
- ✅ 通过 Hook 自动捕获
- ✅ 通过 MCP 工具显式检索
- ✅ 最灵活 — 可以只用一种或两种
- ✅ 两全其美

### 缺点
- ⚠️ 资源使用略高（两个系统同时运行）
- ⚠️ 需要理解的配置选项较多

### 性能影响
- 记忆检索：每次查询 5-30ms
- 存储写入：每次会话结束 2-10ms
- 额外内存：双系统缓存约 5-20MB

---

## 模式二：仅 MCP

**适用于：** 想要完全程序化控制的开发者，或 Hook 不可用的场景。

```json
{
  "plugins": {
    "entries": {
      "unified-memory": {
        "hook": "disabled",
        "config": {
          "mcpEnabled": true
        }
      }
    }
  }
}
```

### 优点
- ✅ 完全控制内存操作
- ✅ 可从外部工具程序化访问
- ✅ 无自动写入 — 一切都是显式的
- ✅ 比双模式资源使用低

### 缺点
- ❌ 无自动捕获 — 必须手动调用 `memory_write`
- ❌ 相同功能需要更多代码
- ❌ 需要开发者集成

### 性能影响
- 只在调用工具时付出成本
- 无后台检索开销
- 内存：约 2-10MB（单系统）

---

## 模式三：仅 Hook

**适用于：** 简单透明运行，从不想操心记忆管理的场景。

```json
{
  "plugins": {
    "entries": {
      "unified-memory": {
        "hook": "enabled",
        "config": {
          "mcpEnabled": false,
          "autoExtract": true
        }
      }
    }
  }
}
```

### 优点
- ✅ 零配置 — 开箱即用
- ✅ 永远不需要修改代码
- ✅ 对用户完全透明
- ✅ 最低复杂度

### 缺点
- ❌ 无法显式搜索或管理记忆
- ❌ 所有召回都是自动的（仅 Hook 注入）
- ❌ 无程序化访问

### 性能影响
- 后台检索仍在运行
- 无 MCP 服务器开销
- 内存：约 5-15MB

## 资源使用对比

| 资源 | Hook + MCP | 仅 MCP | 仅 Hook |
|------|:----------:|:------:|:--------:|
| **内存** | ~15-30MB | ~5-15MB | ~10-20MB |
| **CPU（空闲）** | ~0.5-2% | ~0.1-0.5% | ~0.5-2% |
| **CPU（活跃）** | ~2-5% | ~1-3% | ~2-4% |
| **存储/会话** | ~5-15KB | ~0-10KB | ~5-15KB |
| **延迟开销** | ~15-120ms | ~0-50ms | ~10-80ms |

## 场景推荐

### 在以下情况下使用 Hook + MCP：

- ✅ 你是一个希望记忆"自动工作"的终端用户
- ✅ 你偶尔想搜索旧的记忆
- ✅ 你不确定选择哪种模式
- ✅ 你想要自动捕获但也需要程序化访问

### 在以下情况下使用仅 MCP：

- ✅ 你在统一记忆之上构建应用程序
- ✅ 需要与外部工具或 CI/CD 集成
- ✅ 想要完全控制存储的内容
- ✅ 你的环境中 Hooks 被阻止

### 在以下情况下使用仅 Hook：

- ✅ 追求最大简单性
- ✅ 从不需要手动搜索记忆
- ✅ 不想要任何额外的 UI 或工具
- ✅ 可以接受纯自动召回

---

## 模式间迁移

切换模式很简单 — 所有模式共享相同的记忆存储：

```
Hook + MCP  ←→  仅 MCP  ←→  仅 Hook
     │              │            │
     └──────────────┴────────────┘
              (相同的记忆存储)
```

**切换方法：** 只需更新配置。无需数据迁移。
